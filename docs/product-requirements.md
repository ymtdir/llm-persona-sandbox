# プロダクト要求定義書 (Product Requirements Document)

## プロダクト概要

### 名称

**LLM Persona Sandbox** - 自分以外全員AIの2ch風掲示板アプリケーション

### プロダクトコンセプト

- **「自分以外全員AI」体験**: ユーザーが投稿すると、複数のAIキャラクター(マジレスニキ、煽りカス等)が自動的にレスを生成し、本物の2ch住民のようにスレッドが盛り上がる
- **2ch文化の忠実な再現**: レス番号、アンカー機能、典型的な口調・ネットスラング、キャラクター間の掛け合いなど、2chの雰囲気を技術的に再現
- **コンテンツ創作支援ツール**: YouTubeの2chまとめ動画、小説の壁打ち、創作のアイデア出しなど、エンターテイメント・創作活動を支援する実用的なツール

### プロダクトビジョン

「自分以外全員AI」の2ch風掲示板体験を通じて、一人で気軽に2ch文化を楽しんだり、YouTubeの2chまとめ動画用のコンテンツを著作権クリアな形で生成できるプラットフォームを提供する。好きなテーマで投稿すれば、多様なAIキャラクターが即座に反応し、まるで本物の2chスレッドのような自然な掛け合いが展開される。技術的には、Ollama + Docker ComposeによるゼロコストのローカルLLM運用を実現し、開発環境と本番環境の完全一致による安定運用を目指す。

### 目的

- YouTubeの2chまとめ動画制作者向けに、著作権クリアなオリジナルスレッドを即座に生成できるツールの提供
- 2ch文化を懐かしむユーザーが、一人で気軽に2ch風の掛け合いを楽しめるエンターテイメント体験の提供
- 小説家・創作者向けに、キャラクター同士の掛け合いを試せる壁打ちツールの提供
- Ollamaを用いた複数ペルソナの自動レス生成機能の技術検証
- ローカルLLM（Ollama）によるゼロコストLLM運用の実現可能性検証
- Docker Composeによる開発・本番環境の一致とデプロイの簡素化

## 技術スタック

### フレームワーク・ランタイム

- **Framework**: Hono（超軽量、TypeScript対応）
- **View**: Hono + JSX（サーバーサイドHTML生成）
- **Runtime**: Node.js on Docker
- **Validation**: Zod（型安全なバリデーション）

### LLM

- **開発・本番共通**: Ollama (Docker)
- **モデル**: llama3.1:8b（高速）/ llama3.1:70b（高品質）
- **API**: Ollama HTTP API（互換性のあるOpenAI API形式）
- **コスト**: $0（レンタルサーバー代のみ）

### データベース

- **開発・本番共通**: PostgreSQL (Docker)
- **接続**: パラメータ化クエリによる安全なアクセス
- **永続化**: Dockerボリュームによるデータ保持

### デプロイ

- **コンテナ管理**: Docker Compose
- **構成**: 3サービス（app、ollama、db）
- **開発環境**: Docker Desktop
- **本番環境**: レンタルサーバー（VPS等）+ Docker Compose
- **環境の一致**: `docker-compose.yml`により開発・本番環境が完全一致

### アーキテクチャ図

```
[ユーザー] → ブラウザでスレッドに投稿
       ↓
[Hono + JSX] → サーバーサイドでHTML生成（2ch風UI）
       ↓
[Docker Compose環境]
  ├─ app（Node.js + Hono）
  ├─ ollama（LLMサーバー）
  └─ db（PostgreSQL）
```

## 成功指標(KPI)

### プライマリーKPI

- **面白さ**: テストユーザーの80%以上が「生成されたスレッドが面白い」と評価
- **コンテンツ創作への実用性**: YouTuber・創作者が「動画ネタ/創作の壁打ちに使える」と評価（70%以上）
- **2ch文化の再現度**: 生成されたスレッドが本物の2chスレッドと区別がつかない(テストユーザーの70%以上が「自然」と評価)
- **レスポンス時間**: ユーザー投稿からAIレス生成完了まで10秒以内(3-5キャラクターのレス生成、実用範囲)

### セカンダリーKPI

- **コスト効率**: 1スレッド(20-30レス)生成のLLM APIコストが$0（レンタルサーバー代除く、Ollama使用）
- **キャラクター一貫性**: 同じキャラクターが複数回登場した際、口調・態度が一貫している(80%以上)
- **レス生成品質**: 生成されたAIレスの90%以上がキャラクター設定に一致し、文脈を適切に理解している(人間による評価)
- **技術検証の完了**: 6つの検証ゴール(Ollama API基本接続、ペルソナ切り替え、連続会話、キャラクター間相互作用、2ch的振る舞い、パフォーマンス/コスト検証)をすべて達成
- **環境の一致**: Docker Desktopでの検証環境と本番環境(Docker Compose)が完全に一致し、環境差異によるトラブルがゼロ

## 機能要件

### コア機能(MVP)

#### 1. スレッド作成・表示機能

**ユーザーストーリー**:
ユーザーとして、2ch風のUIで新しいスレッドを作成し、スレッドタイトルと初回投稿を行いたい

**受け入れ条件**:

- [ ] スレッド作成フォームでタイトルと本文を入力できる
- [ ] 作成されたスレッドは一意のIDを持つ
- [ ] スレッド一覧ページで全てのスレッドを閲覧できる
- [ ] スレッド詳細ページで全てのレスが時系列順に表示される
- [ ] 2ch風のレイアウト(MS Pゴシック風フォント、#f0e0d6背景色、レス番号表示)

**優先度**: P0(必須)

#### 2. ユーザー投稿機能

**ユーザーストーリー**:
ユーザーとして、既存のスレッドに対してテキストで返信を投稿したい

**受け入れ条件**:

- [ ] スレッド詳細ページ下部にテキストエリアと送信ボタンがある
- [ ] 投稿内容はデータベースに保存される
- [ ] レス番号が自動採番される
- [ ] 投稿者名は「名無しさん」として保存される
- [ ] 投稿後、スレッドページにリダイレクトされる

**優先度**: P0(必須)

#### 3. AIキャラクター自動レス生成(基本)

**ユーザーストーリー**:
ユーザーとして、自分が投稿した後、複数のAIキャラクターが自動的にレスを生成し、スレッドが盛り上がるのを見たい

**受け入れ条件**:

- [ ] ユーザー投稿後、3-5体のAIキャラクターが自動的にレス生成を開始する
- [ ] 各キャラクターは異なる口調・視点で応答する(マジレスニキ、煽りカス、物知りおじさん、ROM専、新参の5キャラクター)
- [ ] 生成されたレスはデータベースに保存され、スレッドに表示される
- [ ] 生成失敗時はエラーログを記録し、スレッド表示には影響しない
- [ ] 生成されたレスは`is_user_post=0`フラグで区別される

**優先度**: P0(必須)

#### 4. ペルソナ定義・管理機能

**ユーザーストーリー**:
開発者として、AIキャラクターのペルソナ(System Prompt、口調、発言頻度等)を外部ファイルで管理し、容易に調整・追加できるようにしたい

**受け入れ条件**:

- [ ] `src/lib/personas.ts`にペルソナ定義が記述されている
- [ ] 各ペルソナは`id`, `displayName`, `systemPrompt`, `temperature`, `keywords`を持つ
- [ ] ペルソナの追加・編集がコードの変更のみで可能(再デプロイ不要)
- [ ] 最低5種類のペルソナが定義されている

**優先度**: P0(必須)

#### 5. スレッド文脈の管理

**ユーザーストーリー**:
ユーザーとして、AIキャラクターが過去のレスを理解した上で返答し、文脈のある会話を楽しみたい

**受け入れ条件**:

- [ ] AI生成時、スレッドの過去20レスが文脈としてLLM APIに送信される
- [ ] AIレスに「>>1」のようなアンカーが含まれる場合がある
- [ ] 同じキャラクターが複数回登場した際、前回の発言を考慮した返答になる
- [ ] スレッドが長くなった場合(30レス以上)、古いレスは要約または省略される

**優先度**: P1(重要)

#### 6. キャラクター選択ロジック

**ユーザーストーリー**:
ユーザーとして、投稿内容に応じて適切なキャラクターが反応し、自然な掛け合いが生まれることを期待する

**受け入れ条件**:

- [ ] 投稿内容のキーワードに基づいて関心度の高いキャラクターが優先的に選択される
- [ ] 毎回異なるキャラクターの組み合わせが登場する(完全にランダムではない)
- [ ] キャラクターの発言頻度設定が反映される
- [ ] 最低2体、最大5体のキャラクターが選択される

**優先度**: P1(重要)

#### 7. Ollama API連携

**ユーザーストーリー**:
開発者として、Ollamaを用いてコストゼロでLLMによるレス生成を行いたい

**受け入れ条件**:

- [ ] `.env`ファイルの`OLLAMA_BASE_URL`を読み込める（デフォルト: http://ollama:11434）
- [ ] Ollama API（`llama3.1:8b`または`llama3.1:70b`モデル）にリクエストを送信できる
- [ ] Ollamaコンテナが起動していない場合のエラーハンドリングがある
- [ ] APIレスポンスのパース・エラー処理が実装されている
- [ ] 生成されたレスが正常にデータベースに保存される

**優先度**: P0(必須)

#### 8. Docker Compose環境構築

**ユーザーストーリー**:
開発者として、`docker-compose up`一発で全ての環境（アプリ、Ollama、DB）が起動し、即座に検証を開始したい

**受け入れ条件**:

- [ ] `docker-compose.yml`が正しく設定されている
- [ ] アプリケーション、Ollama、PostgreSQLの3サービスが定義されている
- [ ] `docker-compose up -d`でバックグラウンド起動できる
- [ ] ボリュームが正しく設定され、データが永続化される
- [ ] 依存関係（depends_on）が正しく設定されている

**優先度**: P0(必須)

### 将来的な機能(Post-MVP)

#### 9. バッチ生成方式

**ユーザーストーリー**:
開発者として、1回のAPI呼び出しで複数キャラクターのレスを生成し、コストとレイテンシを削減したい

**受け入れ条件**:

- [ ] 1回のLLM API呼び出しで3-5キャラクターのレスをJSON配列形式で取得できる
- [ ] 個別生成方式と比較してAPIコール数が1/3〜1/5に削減される
- [ ] レス品質が個別生成方式と同等以上である
- [ ] 環境変数`USE_BATCH_GENERATION`で切り替え可能

**優先度**: P1(重要)

#### 10. スマートキャラクター選択

**ユーザーストーリー**:
開発者として、LLMを用いてキャラクターの関心度を判定し、より自然な会話を生成したい

**受け入れ条件**:

- [ ] 投稿内容に基づいてLLMが「反応しそうなキャラクター」を選択する
- [ ] 軽量モデル(llama3.1:8b)を使用し、レスポンス速度を優先
- [ ] 環境変数`USE_SMART_SELECTION`で切り替え可能
- [ ] フォールバック機能(LLM失敗時はキーワードマッチング)

**優先度**: P2(できれば)

#### 11. リアルタイムレス表示(SSE)

**ユーザーストーリー**:
ユーザーとして、AIキャラクターがレスを生成する過程をリアルタイムで見たい

**受け入れ条件**:

- [ ] Server-Sent Events(SSE)でAI生成レスがリアルタイムに配信される
- [ ] 「書き込み中...」インジケーターが表示される
- [ ] レスが生成されるたびにページが自動更新される

**優先度**: P2(できれば)

#### 12. 本番環境へのデプロイ

**ユーザーストーリー**:
開発者として、ローカル環境(Docker Desktop)で動作したコードをレンタルサーバー(Docker Compose)に同じ構成でデプロイしたい

**受け入れ条件**:

- [ ] レンタルサーバー（VPS等）にDocker + Docker Composeがインストールされている
- [ ] `docker-compose.yml`を本番サーバーにコピーし、`docker-compose up -d`でデプロイできる
- [ ] 環境変数（`.env`）が本番環境用に設定されている
- [ ] ドメインまたはIPアドレスでアクセス可能
- [ ] データベースとOllamaモデルが永続化されている

**優先度**: P1(重要)

#### 13. パフォーマンス測定機能

**ユーザーストーリー**:
開発者として、1スレッドあたりのコスト、レスポンス時間、トークン数などを測定し、最適化の材料にしたい

**受け入れ条件**:

- [ ] APIコール数、使用トークン数、レスポンス時間がログに記録される
- [ ] 1スレッドあたりの生成時間が計算される
- [ ] バッチ生成と個別生成のパフォーマンス比較ができる
- [ ] 測定結果をCSVまたはJSON形式でエクスポートできる
- [ ] Ollamaのモデル別（8b vs 70b）のパフォーマンス比較ができる

**優先度**: P1(重要)

## 非機能要件

### パフォーマンス

- **レス生成時間**: ユーザー投稿からAIレス生成完了まで10秒以内(3-5キャラクター、llama3.1:8bモデル、CPUまたはGPU環境)
- **ページ表示速度**: スレッド一覧・詳細ページの初回表示が1秒以内(100スレッド、1000レスまで)
- **データベースクエリ**: レス取得クエリが50ms以内(PostgreSQL、1000レスまで)
- **Ollama API呼び出し**: レスポンスタイムが5秒以内(平均、llama3.1:8bモデル)
- **Docker起動時間**: `docker-compose up`でサービス全体が30秒以内に起動

### ユーザビリティ

- **2ch風UI**: MS Pゴシック風フォント、#f0e0d6背景色、レス番号表示など、2chの雰囲気を再現
- **モバイル非対応**: 検証段階のためデスクトップブラウザのみサポート
- **エラーメッセージ**: LLM API失敗時、ユーザーに明確なエラーメッセージを表示(「AIレス生成に失敗しました」等)
- **READMEドキュメント**: 環境構築から実行までの手順が明記されている

### 信頼性

- **データ損失ゼロ**: ユーザー投稿は必ずデータベースに保存される(AI生成失敗時も)
- **AI生成失敗時の挙動**: Ollama API失敗時、スレッド表示は正常に継続し、エラーログのみ記録
- **ロールバック**: データベース書き込み失敗時、トランザクションをロールバック
- **Ollamaコンテナ障害対策**: Ollamaコンテナが停止している場合、明確なエラーメッセージを表示し、再起動を促す
- **データ永続化**: PostgreSQLとOllamaのモデルデータがボリュームで永続化される

### セキュリティ

- **環境変数の管理**: `.env`ファイルに記載し、`.gitignore`で除外
- **XSS対策**: ユーザー投稿内容をサニタイズし、HTMLタグをエスケープ
- **SQLインジェクション対策**: パラメータ化クエリを使用
- **認証機能**: 検証段階では実装しない(誰でもアクセス可能)
- **コンテナ間通信**: Dockerネットワーク内でのみOllama APIにアクセス可能（外部公開しない）

### スケーラビリティ

- **スレッド数**: 1000スレッドまで正常に動作
- **レス数**: 1スレッドあたり1000レスまで正常に動作
- **同時接続数**: 検証段階では1-5ユーザー同時アクセスを想定
- **将来的な拡張性**: Ollamaサーバーのスケールアウト（複数コンテナによる負荷分散）が可能な設計
- **リソース管理**: Docker Composeでメモリ・CPU制限を設定可能

### 保守性

- **コード品質**: TypeScriptで型安全性を担保
- **テスト**: 単体テストは実装しない(検証段階のため)
- **ドキュメント**: コードコメントで主要機能の動作を説明、READMEにDocker Compose起動手順を記載
- **ログ**: Ollama API呼び出し、エラー発生時のログをコンソールに出力
- **コンテナログ**: `docker-compose logs`で各サービスのログを確認可能
- **環境の再現性**: `docker-compose.yml`により、誰でも同じ環境を再現可能

## スコープ外

明示的にスコープ外とする項目:

- **ユーザー認証・ログイン機能**: 検証段階では誰でもアクセス可能
- **管理画面**: スレッド削除、ユーザーBAN等の管理機能は実装しない
- **画像アップロード機能**: テキストのみサポート
- **リアルタイムチャット機能**: WebSocketによる双方向通信は実装しない
- **通知機能**: メール通知、プッシュ通知等は実装しない
- **検索機能**: スレッド・レス検索機能は実装しない
- **レス編集・削除機能**: 投稿後の編集・削除は不可
- **モバイルアプリ**: Webアプリケーションのみ
- **多言語対応**: 日本語のみサポート
- **SEO対策**: 検証段階では不要
- **アクセス解析**: Google Analytics等は実装しない
