## アーキテクチャパターン

### レイヤードアーキテクチャ

```
┌─────────────────────────────────────────┐
│   Webレイヤー (routes/)                 │
│   - ThreadRoutes                         │
│   - HTTPリクエスト処理                   │
├─────────────────────────────────────────┤
│   サービスレイヤー (services/)           │
│   - ThreadManager, PostManager           │
│   - ResponseGenerator                    │
│   - ビジネスロジック                     │
├─────────────────────────────────────────┤
│   データレイヤー (lib/)                  │
│   - DatabaseClient, OllamaClient         │
│   - データ永続化・外部API呼び出し        │
└─────────────────────────────────────────┘
```

**責務分担**:
- **Webレイヤー**: HTTPリクエスト受付、バリデーション、レスポンス生成
- **サービスレイヤー**: ビジネスロジック、データ変換、トランザクション制御
- **データレイヤー**: データ永続化、外部APIアクセス

**詳細設計**: [機能設計書](./functional-design.md#コンポーネント設計)、[リポジトリ構造](./repository-structure.md#レイヤードアーキテクチャ)

## データ永続化戦略

### ストレージ方式

| データ種別         | ストレージ         | 理由                                                 |
| ------------------ | ------------------ | ---------------------------------------------------- |
| スレッド・レス     | PostgreSQL         | ACID準拠、外部キー制約、インデックス高速検索         |
| ペルソナ定義       | TypeScriptファイル | 型安全性、デプロイ時ビルド済み、実行時高速読み込み   |
| Ollamaモデルデータ | Dockerボリューム   | モデルファイル永続化、再ダウンロード回避             |
| 環境設定           | .envファイル       | 機密情報分離、環境ごと設定切替、Gitコミット対象外    |

### データベース設計

**スキーマ詳細**: [データモデル定義書](./data-models.md#データベーススキーマ)

**設計の意図**:
- UUID型: 分散システムでもID衝突なし、URLセーフ
- ON DELETE CASCADE: スレッド削除時に関連レス自動削除
- CHECK制約: データベースレベルでバリデーション
- インデックス: 頻出クエリ最適化（スレッド一覧、レス取得）

**バックアップ**: [デプロイメントガイド](./deployment-guide.md#バックアップ戦略)

## パフォーマンス要件

### レスポンスタイム

| 操作                | 目標時間  | 測定環境                 |
| ------------------- | --------- | ------------------------ |
| ユーザーレス投稿    | 200ms以内 | CPU: 4コア、メモリ: 8GB  |
| AIレス生成（3-5体） | 10秒以内  | llama3.1:8b、CPU推論     |
| AIレス生成（3-5体） | 3秒以内   | llama3.1:8b、GPU推論     |
| スレッド一覧表示    | 500ms以内 | 100スレッド              |
| スレッド詳細表示    | 1秒以内   | 1000レス                 |

### リソース使用量

| リソース                | 上限  | 理由                                         |
| ----------------------- | ----- | -------------------------------------------- |
| アプリメモリ（app）     | 512MB | Node.js + Hono軽量                           |
| DBメモリ（db）          | 256MB | PostgreSQL、10000レスまで十分                |
| Ollamaメモリ（ollama）  | 8GB   | llama3.1:8bモデルロード（5GB）+ 推論（3GB）  |
| ディスク（ollama-data） | 10GB  | モデルファイル（4.7GB）+ 余裕                |
| ディスク（db-data）     | 1GB   | 10000レスで約100MB + 余裕                    |

### 最適化戦略

1. **データベースクエリ最適化**: インデックス活用、LIMIT句
2. **AIレス生成の非同期処理**: ユーザー体験優先、バックグラウンド実行
3. **スレッド履歴制限**: 最新20件のみ送信（トークン数削減）
4. **Ollamaモデルプリロード**: 起動時自動ロード、初回遅延回避

**詳細**: [機能設計書](./functional-design.md#パフォーマンス最適化)

## セキュリティアーキテクチャ

### データ保護

- **環境変数管理**: .envに機密情報、.gitignoreで除外
- **コンテナ間通信制限**: Ollama・PostgreSQLは内部ネットワークのみ
- **ファイルパーミッション**: .env は chmod 600（所有者のみ読み書き）

### 入力検証

- **バリデーション**: Zodによる厳格なスキーマ検証
- **サニタイゼーション**: Hono JSXの自動HTMLエスケープ
- **SQLインジェクション対策**: パラメータ化クエリ（$1, $2形式）

**実装例**:
```typescript
// 安全な実装
await db.query('SELECT * FROM posts WHERE thread_id = $1', [threadId]);

// 危険な実装（NG）
await db.query(`SELECT * FROM posts WHERE thread_id = '${threadId}'`);
```

### エラーハンドリング

- **セキュアなエラー表示**: 詳細をユーザーに隠蔽、ログに記録
- **本番環境**: スタックトレース非表示

**詳細**: [機能設計書](./functional-design.md#エラーハンドリング)

## スケーラビリティ設計

### データ増加への対応

- **想定データ量**: スレッド1000件、レス10000件、DB容量約100MB
- **対策**: インデックス、LIMITによるページネーション
- **将来**: 古いスレッドのアーカイブ（6ヶ月以上前）

### 機能拡張性

- **環境変数による設定切替**:
  - `OLLAMA_MODEL`: llama3.1:8b / llama3.1:70b
  - `USE_BATCH_GENERATION`: true / false
  - `USE_SMART_SELECTION`: true / false
- **ペルソナ追加**: personas.ts編集で新キャラクター追加

### スケールアウト（将来実装）

- **Ollama水平スケール**: Nginx負荷分散、複数コンテナ
- **DBレプリケーション**: マスター（書込）+ スレーブ（読込専用）

## 技術的制約

### 環境要件

- **OS**: Linux / macOS / Windows（WSL2）
- **最小メモリ**: 16GB（CPU推論）、12GB（GPU推論）
- **ディスク容量**: 20GB
- **必須ソフト**: Docker Engine 24.x、Docker Compose 2.x

### パフォーマンス制約

- **CPU推論**: 1キャラ2-3秒、3-5キャラで10秒程度
- **GPU推論**: 1キャラ0.5-1秒、3-5キャラで3秒程度
- **モデル切替**: llama3.1:8b→70bで10-30秒ロード時間

### セキュリティ制約（MVP段階）

- 認証機能なし（誰でもアクセス可能）
- HTTPS非対応（HTTP通信のみ）
- レート制限なし（DDoS対策なし）

**本番環境推奨**: Nginx + HTTPS、認証機能追加

## 依存関係管理

| ライブラリ        | バージョン管理 | 理由                                   |
| ----------------- | -------------- | -------------------------------------- |
| hono              | ^4.x           | マイナーバージョンアップは互換性維持   |
| zod               | ^3.x           | 安定版、破壊的変更リスク低             |
| pg                | ^8.x           | 長期安定版、PostgreSQL 17.x互換        |
| typescript        | ~5.3.0         | パッチバージョンのみ自動更新           |

**更新戦略**:
1. `npm outdated` で更新確認
2. マイナーバージョンは `npm update` で自動
3. メジャーバージョンは手動更新 + テスト

## 監視・ログ

### ログ出力

**フォーマット**:
```
[2025-01-15T14:32:18.123Z] [INFO] ThreadManager: Thread created (id: abc123)
[2025-01-15T14:32:20.456Z] [ERROR] OllamaClient: Connection failed
```

**ログレベル**:
- ERROR: システムエラー（DB接続失敗、Ollama接続失敗）
- WARN: 一時的エラー（LLMタイムアウト）
- INFO: 正常動作（スレッド作成、レス投稿）

**確認方法**:
```bash
docker-compose logs -f app
```

**詳細**: [デプロイメントガイド](./deployment-guide.md#監視・メンテナンス)
